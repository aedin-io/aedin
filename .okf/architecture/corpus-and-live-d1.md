# The projection

The live site serves a **projected subset** of the corpus, not the corpus itself:

- `web/scripts/build-d1.cjs` reads [`aedin.sqlite`](/architecture/db-split.md) and emits `web/d1/data.sql` — only **served** entities, only **projected columns**. The served set = entities with `scope_tier` set **UNION any entity referenced by an `ai_reviewed` claim/trait** (plus slugged merge-tombstones). **The real serving gate is the `slug`**, not the row: `entity/[slug].astro` is on-demand SSR (`prerender=false`, reads the D1 binding at request time), so a row with NULL `slug` is on D1 but **page-less** — and a newly-slugged row resolves live with **no Pages rebuild** (pure D1-data publish).
- That is published to Cloudflare **D1** (`database_name=agroeco`) via `wrangler d1 execute --remote`, and the Astro Pages site is deployed with `npm run deploy`.

Consequences:

- The D1 `entities`/`claims` tables are a **served subset** and OMIT quarantined/removed rows and unserved entities. Columns absent from the projection (e.g. `source_table`) **do not exist on live D1** — query live D1 by projected columns (`variety_type`, `parent_entity_id`) only.
- **Corpus and live D1 routinely drift.** Local `scope_tier` changes do not appear live until a rebuild + republish. A full rebuild surfaces everything served-in-corpus at once.

# Publish hazards

- **Data publish ≠ Pages deploy.** `wrangler d1 execute --remote` (data) is separate from `npm run deploy` (Pages-only).
- **Ordering hazard:** a worker that SELECTs a column must NOT deploy before that column exists on live D1, or `/entity/*` 500s. Add the column (remote `ALTER TABLE`) before deploying code that reads it.
- Surgical patches (remote `ALTER`/`UPDATE` for a single column/subset) are used to avoid republishing a concurrent session's in-flight corpus work.
- **Surgical variety republish:** `web/scripts/gen-variety-delta.cjs` emits a delta for a set of served-variety slugs — the variety `entities` rows plus their synthesized inherited `entity_trait_claims` (it replicates `build-d1.cjs`'s projection and the `vid*1e9+row.id` inherited-id scheme) — so new/re-slugged varieties go live without a full `data.sql` republish. It writes `INSERT … ON CONFLICT(id) DO UPDATE` for entities (an id already live with a NULL slug is re-slugged, not a PK error) and `INSERT OR IGNORE` for trait claims, making the delta **idempotent / safely re-runnable**. Apply via `wrangler d1 execute --remote --file`. A variety whose **parent species is not served** must be excluded (its parent-link would 404) until the parent is published.
  - **Inheritance is a DERIVED, D1-ONLY product** (materialized at build, never stored in the corpus) — so live `entity_trait_claims` is **larger than the corpus** by the inherited count, *by design* (not a divergence bug; see memory `variety-trait-inheritance`). Only `inheritanceClass==='conserved'` traits inherit (`PLANT_CONSERVED` = climate/envelope only; foundational traits are divergent). Because surgical traits-patches skip inheritance and `gen-variety-delta` is additive + per-slug-set, **live inheritance drifts behind the corpus** over time. **`web/scripts/gen-inheritance-refresh.cjs`** (TDD'd) reconciles it globally: it reuses `build-d1.cjs::selectReadSubset` (the SAME materialization the full build uses — no duplicated scope/id logic), isolates the inherited subset, and emits a **full-replace of just that subset** — `INSERT OR IGNORE` the variety entities/sources, then `DELETE FROM entity_trait_claims WHERE inherited_from_entity_id IS NOT NULL` + re-INSERT (base + GloBI untouched; no stale/orphans; deterministic ids → idempotent). Apply via `wrangler d1 execute agroeco --remote --file`. First run 2026-06-28 (9,736→10,378 inherited / 855→950 varieties). **It is a MANUAL tool** — but a **PostToolUse Bash reminder hook** (`.claude/hooks/posttooluse-d1-inheritance-reminder.mjs`, wired in `.claude/settings.json`) fires after any `wrangler d1 execute --remote` trait publish and reminds to run it, so the drift is surfaced at the moment it's introduced rather than silently forgotten (it stays silent on the refresh's own patch + non-trait publishes). No cron/auto-apply (production writes stay deliberate).
  - **Pathogen transmission traits are also a DERIVED, D1-ONLY product** (`backend/lib/transmission-traits.js::derivePathogenTransmission`, materialized in `selectReadSubset` alongside inheritance, TDD'd). For each served pathogen that is the object of ≥1 `ai_reviewed` `disease_vector` edge it emits `transmission_vector` (value_json list of the pathogen's vector names) + `transmission_mode='vector_borne'` (value_text), with synthetic ids `pathogenId*1e9+offset` and `inherited_from_entity_id=null` (so the inheritance refresh never deletes them). This is the **biological half only** — abiotic `transmission_mode` (seed/soil/sap/water/…) is extracted, a follow-on. Regenerated on every full build; **live surfacing rides the next full `data.sql` republish**, not a surgical patch. See memory `vector-as-pathogen-misencoding`.
- **Surgical claims patch** (two variants, for when promoted `ai_reviewed` literature claims outrun the last publish without touching the in-sync GloBI tier):
  - `web/scripts/build-d1-patch.cjs` — **source-scoped**: `DELETE FROM claims WHERE source_id IN (…)` then re-inserts all of those sources' claims + entities + verdicts. For **re-ingestion** of a source (needs the `reset-sources-for-reingest.js` backup JSON for orphaned-verdict cleanup).
  - `web/scripts/gen-claims-patch.cjs` — **claim-id-scoped, additive (no DELETE)**: computes its own delta (pulls the live `ai_reviewed` claim-id set via `wrangler`, diffs against local) and emits only the missing claims plus their not-yet-live entities (`INSERT OR IGNORE`), sources (`OR REPLACE`), and verdicts. For **new promotions scattered across sources**. Literature region rides `claims.regional_context` (a projected column), so **no `claim_localities`** are needed — that table is GloBI-only. **Page-less guard:** it never emits a claim whose subject/object would reference a not-yet-live, slug-less entity (a broken `/entity/<slug>` page — typically a `needs_dedup` duplicate awaiting variety-intake); such claims are held + reported rather than shipped.
- **Serving the page-less referenced tail** (2026-06-29) — the publish backlog is gated on **entity serving, not claim publishing**: `gen-claims-patch` HOLDS a claim whose entity is page-less. `serve-referenced-entities.js` ([entities](/datasets/entities.md)) gives the clean literature-ingested referenced entities a slug + `scope_tier=0`. Then a **dedicated entity patch is required, not just `gen-claims-patch`**: `web/scripts/gen-served-entities-patch.cjs` emits **`INSERT OR REPLACE INTO entities`** for the just-served set (ids from a serve-script backup or `--revlog-since`). This is the keystone — `gen-claims-patch` uses `INSERT OR IGNORE` and emits entities only when **not already live**, so it CANNOT fix the (majority) case of an entity already on D1 as a NULL-slug row; `OR REPLACE` overwrites those rows' slug/`scope_tier`. Publish order: **entities patch → claims patch → traits patch** (a claim must not reach D1 before its entity's slug). A full `build-d1` rebuild is NOT an option — it would wipe the ~212K live-only `tier2_globi` claims; surgical only.
- **Surgical traits patch** (two variants, symmetric with the two claims patches):
  - `web/scripts/build-d1-traits-patch.cjs` — **full-table**: republishes the whole `ai_reviewed` `entity_trait_claims` set (`INSERT OR REPLACE`), with a **page-less guard** and an optional **`--served-since=<iso>` scope**. Its entity writes are `INSERT OR REPLACE INTO entities` across ALL trait-referenced entities. **Use only when a full trait refresh is intended** — in a multi-chat corpus the entity REPLACE can overwrite live rows and leak a concurrent chat's un-published entity edits (e.g. the dedup chat's held `merged_into_entity_id`).
  - `web/scripts/gen-traits-patch.cjs` (added 2026-07-03) — **additive-delta**, the trait-table sibling of `gen-claims-patch.cjs`: pulls the live `ai_reviewed` trait-id set (`id < 1e9`, excluding the build-time inherited rows) via `wrangler`, diffs against local, and emits only the missing traits + their **not-yet-live** entities (`INSERT OR IGNORE`), sources, and verdicts. **Prefer this for new-promotion deltas** — it touches ZERO already-live entity rows, so it cannot leak held entity work. Same page-less guard. This is the safe default when other chats are editing the shared corpus.
