# sim_params layer

**sim_*** is a regenerable, **corpus-local** schema layer for the polyculture/agroforestry simulator — five typed tables holding DERIVED and DESIGNED modelling artifacts. It is kept strictly separate from the provenance-gated `claims` and `entity_trait_claims` so the corpus retains its audit trail and fact-based discipline: sourced trait facts live in the corpus; derived/designed simulation parameters live here.

## Design principle

- **Sourced trait facts** live in the corpus (`entity_trait_claims`, `claims`) — provenance-gated, citable.
- **Derived/designed model params** live in `sim_*` — computed or hand-curated, tagged with a `param_status` envelope, never citable as a fact.
- This boundary lets a simulator consume rich modelling parameters without diluting the corpus's role as a citable knowledge base.

## Provenance policy — sim inputs may be sourced from anywhere (2026-07-03)

Values that feed ONLY the simulator — the `sim_*` rows and the scalar inputs the derivation reads (e.g. `entities.maximum_height_cm`, `entities.spread_cm`) — may be sourced from **any** reference, **including copyrighted books**, because the simulator consumes only the factual VALUE (a number) and never surfaces, cites, shows, or distributes the original source. Facts aren't copyrightable; only a source's expression/compilation is. Store such values in an **un-surfaced** place (the `entities` scalar attribute, shown as a bare number with no citation), NOT as a cited `entity_trait_claims` row that would display the source on the public entity page. The citable corpus (`entity_trait_claims`, `claims`) is UNCHANGED — surfaced claims remain bound by the facts-only copyright/quote policy. The distinction is **surfacing**: shown-with-source → copyright applies; anonymous sim input → it doesn't. First application: backfilling `entities.maximum_height_cm` from the in-repo Rubatzky text so the height-gated `sim_plant_growth.param_status='derived'` label lights up.

## Un-surfaced sim-trait source cascade (USDA → Trefle)

The un-surfaced scalar inputs `entities.{maximum_height_cm, growth_habit, min_root_depth_cm}` are backfilled by a **pluggable, priority-ordered source cascade** — separate from the provenance-gated `entity_trait_claims` ingestion. Each source is a thin adapter `{name, cacheDir, extract(record)→{query_name, fields}}` registered in priority order in `backend/lib/sim-trait-sources.js`. A fetcher per source caches raw records to `backend/<source>-cache/` (gitignored, regenerable); the runner `backend/backfill-sim-traits.js` iterates the cascade and applies each source through the shared **fill-if-NULL** write-core `backend/lib/sim-scalar-backfill.js`, so a higher-priority source always wins and lower sources fill only what's left NULL. The order is pure config — reorderable, and BIEN / monograph adapters drop in later without touching the runner.

- **Source #1 — USDA PLANTS** (`fetch-usda-characteristics.js` + `lib/usda-normalize.js`): public-domain REST. Height from `Height, Mature (feet)`, min root from `Root Depth, Minimum (inches)` (both in `PlantCharacteristics`), growth habit from `PlantProfile` `Durations`+`GrowthHabits` (NB: these are NULL on search results — a `PlantProfile`-by-symbol call is required). Name-matched by accepted binomial, ambiguity→skip.
- **Source #2 — Trefle** (`fetch-trefle-sim.js` + `lib/trefle-normalize.js`): live REST, token in `.env`. Contributes growth habit (`specifications.growth_habit`) + occasional min root depth / height (all already in cm). Stored `trefle_id`s are stale → resolved by **name** with binomial verification.

Provenance stays un-surfaced: the write-core logs each change to `revision_log` with source-agnostic `changed_by='sim-trait-backfill'` / `method='sim_trait_backfill'` (no source named in a surfaced field); per-source attribution lives only in the un-surfaced backup JSON under `backend/backups/`. Every write is fill-if-NULL, plantae-only, sanity-range-guarded, and reversible. USDA's mature-height coverage skews to woody/field crops (the long tail of tropical/cultivar taxa has none), so height flips fewer rows than growth-habit fills — the cascade's chief value on the designed-default majority is cleaner life-form classification.

**Extended fields (migration 074).** The USDA adapter also extracts categorical ecosystem-service inputs (`nitrogen_fixation`, `cn_ratio`, `growth_rate`, `fertility_requirement`) and plant **soil-tolerance facts** (`soil_texture_adaptation`, `anaerobic_tolerance`, `caco3_tolerance`, `salinity_tolerance`, `drought_tolerance`, `moisture_use`, `optimal_ph_min/max`) onto `entities`. The service inputs feed `sim_ecosystem_service`; the tolerance facts are **banked un-surfaced for the future site soil-suitability / plantability feature** (which will match plant tolerances × a site soil profile with horizon depths — a separate build). **Matcher fix (durable):** USDA returns `AcceptedId=0` for all search rows, so `matchAccepted` now prefers the single **bare species-level** row (one 2-word italic group) over a crop's infraspecific taxa sharing the binomial — without it, major crops with USDA subspecies (corn/tomato/brassicas) were silently ambiguity-skipped, halving coverage for both this layer and the height cascade.

## The tables

The four core tables carry a shared **provenance envelope** (`sim_ecosystem_service` carries the same envelope):

```
param_status       TEXT NOT NULL CHECK(param_status IN ('derived','designed','override'))
                     — 'derived'  = computed from corpus facts
                     — 'designed' = hand-curated / model-fit constant
                     — 'override' = human/runtime override — NEVER touched by regeneration
derivation_method  TEXT  — pipeline/method name (e.g. a lib/sim-derive.js function)
model_ref          TEXT  — versioned model/constant-set reference (e.g. GROWTH_MODEL_REF)
inputs_json        TEXT  — JSON-encoded inputs the derivation consumed (audit trail)
confidence         TEXT CHECK(confidence IN ('high','medium','low'))
generated_at       TEXT NOT NULL DEFAULT (datetime('now'))
generated_run_id   TEXT  — run tag from derive-sim-params.js, for traceability
```

### `sim_plant_growth` — entity-keyed

Species-level growth-curve parameters. Key: `entity_id` (UNIQUE).
Columns: `life_form`, `time_unit` (`days`|`years`), `max_height_cm`, `max_spread_cm`, `max_root_depth_cm`, `min_root_depth_cm` (migration 073), `root_pattern`, `days_to_maturity`, `height_curve_model`, `height_inflection`, `height_rate_k`, `spread_curve_model`, `spread_inflection`, `spread_rate_k`, `canopy_layer`, `seasonality`, `light_extinction_coeff`, + envelope.

**Root depth is a `[min, max]` range the simulator samples within.** `max_root_depth_cm` is the designed per-life_form ceiling (`LIFE_FORM_DEFAULTS`). `min_root_depth_cm` is sourced from `entities.min_root_depth_cm` (a USDA/Trefle minimum root depth) where available, else `round(MIN_ROOT_DEPTH_FRACTION × max)` (`sim-defaults.js`). The invariant `min ≤ max` is enforced — a sourced min exceeding the designed max lifts the max to `ceil(min × 1.25)`. `inputs_json.min_root_depth_cm.source` records `fact` vs `designed`; the `param_status='derived'` gate stays height-only (a sourced root min raises `confidence`, it does not by itself mark a row `derived`). Bumping the derivation model bumped `GROWTH_MODEL_REF` → `growth_v2`.

### `sim_pest_dynamics` — entity-keyed

Pest/pathogen population-dynamics parameters. Key: `entity_id` (UNIQUE).
Columns: `generations_per_year`, `onset_season`, `onset_months`, `pressure_buildup_rate`, `peak_pressure`, `overwintering`, + envelope.

### `sim_biocontrol` — claim-keyed

Biocontrol efficacy parameters, one row per enemy→pest interaction claim (the only table keyed off `claims`, not `entities`, since efficacy is a property of the relationship). Key: `claim_id` (UNIQUE).
Columns: `enemy_entity_id`, `pest_entity_id`, `control_magnitude`, `response_lag_days`, `establishment`, `specificity`, + envelope.

### `sim_visual` — entity-keyed

Visual/3D-model display parameters for the simulator. Key: `entity_id` (UNIQUE).
Columns: `model_archetype`, `canopy_shape`, `foliage_color`, `produce_color`, `height_scale_cm`, `spread_scale_cm`, + envelope.

### `sim_ecosystem_service` — entity-keyed (migration 074)

Per-plant **soil/nutrient/carbon ecosystem-service indicators** ("Bucket B") for a gardener-facing polyculture *complementarity* view. Key: `entity_id` (UNIQUE). Each indicator is a **categorical qualitative** value (never a flux number) derived from sourced `entities` facts + designed thresholds (`sim-defaults.js` `ES_MODEL_REF='es_v1'`), by the pure `lib/sim-ecosystem-service.js::ecosystemServiceParams`.
Columns: `nitrogen_fixation_class`, `residue_decomposition` (fast/balanced/slow_immobilizing, from C:N), `nutrient_demand`, `rooting_niche` (shallow/medium/deep), `growth_strategy`, `ground_cover`, `life_cycle_class`, `biomass_contribution`, `soil_functions` (a JSON *set* — n_fixer/deep_rooter/ground_cover/perennial_builder/high_biomass — the `soil_health_function` equivalent), + envelope.

**Honesty constraints (from an adversarially-verified research pass — see `docs/sim-ecosystem-services-gap-analysis.md`):** NO single rolled-up "sustainability score" (services trade off; ~75% of traits are multi-service); `residue_decomposition` is a *decomposition* signal, NOT carbon sequestration; `biomass_contribution` is a *potential* proxy (always low-confidence/`source:'designed'`); NO "dynamic accumulator" indicator (popular but zero primary evidence); `confidence` per row from the count of sourced inputs. Bucket A (living/regulating services — pest regulation, pollinator support, facilitation) is a deferred follow-on sourced from the interaction web (`claims`), not this table.

Each table has an index on its key column (`sim_biocontrol` additionally indexes `pest_entity_id`). All `CREATE TABLE`/`CREATE INDEX` statements are `IF NOT EXISTS` — the migration is idempotent.

## Derivation is one-directional: corpus → sim_*, never back

`backend/derive-sim-params.js` is the sole populator. It:

1. **Reads** the corpus — `entities` (for the served plant/pest population and un-surfaced scalar fallbacks like `maximum_height_cm`/`spread_cm`/`min_root_depth_cm`/`growth_habit`), `entity_trait_claims` (`review_status='ai_reviewed'` sourced trait values), and `claims` (`review_status='ai_reviewed'` interactions: `pest_pressure`/`herbivory` for pest population, `biocontrol` for the enemy↔pest edges).
2. **Derives** via the pure functions in `backend/lib/sim-derive.js` (`classifyLifeForm`, `growthCurveParams`, `visualMapping`, `pestDynamics`, `biocontrolDefaults`) — these fall back to the designed constants in `backend/lib/sim-defaults.js` (`LIFE_FORM_DEFAULTS`, `CONTROL_MAGNITUDE`, `BIOCONTROL_DEFAULTS`, `PEST_DEFAULTS`, `canopyLayerForHeight`) wherever a sourced fact is missing.
3. **Writes ONLY the `sim_*` tables** — it never mutates `entities`, `claims`, or `entity_trait_claims`. The corpus is read-only from this script's perspective.

Each regeneration run **full-replaces** `derived`/`designed` rows (`DELETE … WHERE param_status IN ('derived','designed')` then re-`INSERT`) per table, while any row a human hand-set to `param_status='override'` is looked up first and preserved untouched — `writeTable()` skips inserting a derived row over an overridden key. This makes the layer safely regenerable at any time without clobbering manual corrections.

Run modes: dry-run by default (prints would-write counts only); `--apply` performs the write inside a single transaction; `--run-id=<tag>` labels the run (defaults to a timestamp). Every apply logs one `revision_log` row per table (`target_type='sim_layer'`, `method='derive_sim_params:<runId>'`, before/after row counts) via `lib/revision-log.js::logRevisions`, so each regeneration is auditable even though the target isn't a corpus table.

## Not served to live D1

This layer is **corpus-local only**. `web/scripts/build-d1.cjs` does not project any `sim_*` table into `web/d1/data.sql`, and none of the four tables exist on live Cloudflare D1. The public academic site ([corpus vs live D1](/architecture/corpus-and-live-d1.md)) serves citable claims/entities, not designed simulation parameters — the polyculture/agroforestry simulator that consumes `sim_*` is a separate downstream product/repo, reading the corpus SQLite directly (or a future export of it), not the D1 projection.
