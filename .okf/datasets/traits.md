# Trait vocabulary

`traits_vocabulary` is the controlled vocabulary of every trait an entity can carry; `entity_trait_claims` holds the typed, sourced values that reference it. See [/datasets/entities.md] (the entities the claims attach to) and [/datasets/claims.md] (the interaction-claim sibling).

## Schema model
- `traits_vocabulary` (migration 033): `trait_name` (PK), `value_kind` ∈ {numeric, categorical, range, list, boolean}, `expected_unit`, `applicable_bio_categories` (JSON array), `enum_values` (JSON array, for categorical/list), `description`, `introduced_at`.
- `value_kind` drives storage via `backend/lib/trait-value.js` (`encodeTraitValue`/`validateTraitValue`/`decodeTraitValue`): numeric→`value_numeric`, categorical/boolean→`value_text`, range/list→`value_json`. Adding a trait is data-only — no engine change.
- Migrations extend the vocabulary additively: 041 (agronomic_uses), 042 (non-plant traits), 059 (evolvability), 068 (GRIN produce/growth + reproduction + nutrient), **069 (foundational crop traits)**, **070 (edible_part += corm, rhizome)**, **071 (edible_part += bud, inflorescence, calyx)**.

## Foundational crop-trait batch (migration 069, 2026-06-27)
27 plantae traits across five groups — growth form & physiology, environment tolerances, phenology/flowering control, propagation & cultivation system, spacing geometry & nutrient offtake. **Define-only**: the rows define the vocabulary; per-species population is a separate downstream pass (monograph/curated-table sourced, distinct from GRIN narrative extraction). Sourced from a horticulturist gap analysis plus a second pass against Adams "Principles of Horticulture" (which surfaced the perennial/propagation axis — propagation_method, rootstock, seed dormancy, chilling vs vernalization, juvenility, protected culture — that the field-crop monographs miss).

### Populate-time enum refinement (migration 070, 2026-06-28)
The multi-critic gate is also the *vocabulary's* feedback loop: populating `edible_part` from Rubatzky *World Vegetables* surfaced that the 10-value enum lacked tokens for two storage organs, so taro/water-chestnut (a **corm**) and ginger/turmeric/arrowroot/canna/sweet-flag (a **rhizome**) were force-fit to `tuber`/`stem`/`root` and the critics held them. Migration 070 adds `corm` + `rhizome` (enum → 12 values), and the held force-fit claims were curated-corrected to the right organ and re-promoted. **Migration 071** closes the same gap for reproductive organs — `bud` (Brussels sprout, caper, myoga), `inflorescence` (broccoli, cauliflower curd, artichoke), `calyx` (roselle) — all previously force-fit to `flower` (enum → 15 values); 5 promoted + 2 held claims were corrected. `flower` is now reserved for a true edible flower (borage, nasturtium, squash blossom). The pattern — *real data run through the vocabulary exposes a gap, the gap feeds back into the schema* — is the define-then-populate loop closing on itself.

### Population data-hygiene (2026-06-28)
Two recurring extraction errors the multi-critic gate surfaced during monograph population are now guarded in `.claude/agents/trait-table-extractor.md` ("Common value pitfalls"): (1) **woody plants mislabeled `life_cycle=herbaceous_perennial`** — a shrub/subshrub/woody-vine/tree is `woody_perennial` even when the source says only "Perennial" (rosemary/lavender/sage/thyme/hyssop/savory, caper, lemon verbena, vanilla, cassava, pigeon pea); (2) **corm/rhizome collapsed into `tuber`/`stem`/`root`** (see migration 070). Held claims for both classes were curated-corrected + re-promoted.

**Chunk-overlap dedup (`backend/dedup-entity-trait-claims.js`, 2026-06-28):** unchunked-vs-chunked monograph passes re-extract the same per-species value, duplicating `entity_trait_claims` rows. The dedup pass (TDD'd, dry-run-default, backs up deleted rows) distinguishes three cases on the key (entity, trait, value, source): **EXACT dup** (all four equal) → collapse to one (keep-rule: prefer a row with a `source_quote`, tie-break lowest id); **cross-VALUE conflict** (same entity+trait, differing values — e.g. *Ananas* ph_max [6.0, 8.0], celery edible_part granularity) → FLAG to a report, never auto-resolve; **multi-SOURCE same value** → KEEP (corroboration is provenance). First run removed **223 exact dups** (37,749 → 37,526) and flagged **278 cross-value conflicts**.

**Conflict resolution (`backend/resolve-trait-conflicts.js`, 2026-06-28):** a second TDD'd pass resolves the conflicts that have a clear WINNING VALUE an actual source asserted (never fabricated) and deletes only the losing-value claims (multi-source corroboration of the winner kept): list → keep the *superset* value (else flag partial-overlap); numeric → keep one if spread ≤20% (else flag far); boolean → keep the valid {true,false}, drop junk (else flag); categorical → `pest_mobility {none,sedentary}`→sedentary, or keep the substring-richest (else flag). First run resolved 167 groups (**−201 rows**, 37,526 → 37,325). **Source-aware flagging (refined 2026-06-28):** a disagreement is only flagged when a *single source* reported >1 distinct value for the entity+trait (internal extraction inconsistency); a value that differs purely *across* sources is legitimate provenance — two papers, two readings — and is KEPT, never flagged (`hasSameSourceConflict`). This split the original 111 into **50 legitimate cross-source disagreements** (kept) and **58 same-source inconsistencies** flagged for curated/variety review (`backups/trait-conflicts-flagged-*.json`): growth_habit free-text variants, edible_part variety-granularity (celery petiole-vs-root), numeric (*Ananas* ph), tomato-landrace produce_shape. **Remaining backlog:** the 58 same-source inconsistencies (need domain/variety judgment — kept flagged, not auto-resolved, to avoid fabricating a winner).

**Keep-all-values policy (2026-06-28, decision):** distinct trait VALUES are never deleted as "conflicts" — the corpus keeps every sourced data point; aggregation happens at the **serving layer**, not by destroying corpus data. Concretely: `resolve-trait-conflicts.js` no longer touches **numeric** traits at all (two numbers for one entity+trait are range endpoints — a source may state an optimal range the extractor split, or two sources may differ — not a contradiction; 113 previously-deleted numeric values, mostly pH, were restored). The resolver now only collapses genuinely-subsumed values (a list subset inside a superset, a categorical substring) and removes invalid boolean junk. The entity-page **claims list shows every value** (`EntityClaimsTable`); a sidebar **range/envelope** aggregates them for display — `web/src/lib/trait-summary.ts::summarizeTraits` (numeric → min–max, list → union of members, categorical → distinct values) rendered by `web/src/components/TraitSummary.astro` (a panel distinct from `EntityEnvelope`, which reads pre-aggregated `entities.*` scalar columns). Same-source numeric "inconsistencies" are NOT treated as errors for this reason. See [/services/web-site.md].

**Salvage caveat (learned 2026-06-28):** when correcting a *staging payload's* list `value_json`, store an **array**, not `JSON.stringify(array)` — `promoteEntityTraitRow` runs the value through `encodeTraitValue`, which JSON-stringifies it, so a stringified input becomes **double-encoded** (`"[\"corm\"]"`). The corm/rhizome salvage hit this on 15 claims (detected as `unparseable_list` conflicts, fixed by unwrapping one layer).

**Durable design principle:** each trait's `description` carries its *overlap resolution* — how it differs from the existing trait it is most likely confused with (frost_hardiness vs tolerance_temp_min; chilling_requirement_hours vs vernalization_requirement; n/p/k_removal vs nutrient_demand; life_cycle vs growth_habit/growth_determinacy; requires_rootstock conditions height/spacing). The vocabulary cannot enforce these distinctions, so the description is their durable record for populate-time extractors and human curators. The micronutrient axis is modeled as *sensitivity* (the extended `deficiency_sensitivity` enum), not requirement/removal.
